# UX-Core Cognitive Biases Skill

## Purpose

This skill helps analyze UX problems, flows, interfaces, and copy through the lens of cognitive biases from UX-Core. It should be used selectively and only when bias-aware thinking can improve comprehension, trust, decision quality, or behavior in a user-centered and ethical way.

## Core Principle

Do not force bias analysis into every task.
Use this skill only when the user problem involves:

- choices
- attention
- perception
- memory
- trust
- motivation
- uncertainty
- risk
- commitment
- comparison
- conversion

## Use this skill for

- UX audits
- conversion flow reviews
- onboarding reviews
- checkout or payment flows
- pricing and plan comparison
- dashboards and decision-heavy interfaces
- form design
- copywriting review
- empty states, alerts, warnings, confirmations
- experiment ideation
- behavioral friction analysis
- prioritizing improvements based on how users actually think

## Do NOT use this skill for

- purely cosmetic UI suggestions
- code-only refactors
- animation-only requests
- generic brainstorming without a UX objective
- tasks where standard usability principles are sufficient without cognitive-bias framing

## Working Method

When invoked, follow this sequence:

1. Clarify the UX objective
   Identify what the user is trying to achieve:

- understand
- choose
- trust
- complete
- compare
- remember
- return
- act

2. Decide if bias analysis is useful
   Only continue if cognitive bias framing adds meaningful value.

3. Select only the most relevant biases
   Usually use 3 to 7 biases maximum.
   Never dump a long list.

4. Translate each bias into UX language
   For each selected bias, explain:

- why it matters here
- how it may appear in the current experience
- the design risk or opportunity
- a practical design action

5. Prioritize ethically
   Avoid manipulative dark patterns.
   Recommend interventions that improve:

- clarity
- confidence
- decision quality
- user autonomy
- accessibility

6. Format output for action
   Always end with:

- priority recommendations
- suggested experiments or validations
- risks to watch

## Output Formats

Depending on the task, use one of these formats:

### A. Audit Format

- UX objective
- Is bias analysis useful: yes/no
- Relevant biases
- Observed issues
- Design implications
- Recommended changes
- Priority

### B. Ideation Format

- UX objective
- Relevant biases
- Opportunity ideas
- Expected impact
- Validation method

### C. Copy Review Format

- UX objective
- Relevant biases
- Copy issue
- Why it may fail cognitively
- Better alternative
- Notes on clarity and tone

### D. Experiment Format

- Hypothesis
- Bias involved
- UI change
- Expected behavioral outcome
- Metric to track
- Risk / counter-signal

## Preferred Bias Families

Use these more often because they tend to be practical in product work:

- Hick’s Law
- Serial position effect
- Peak-end rule
- Loss aversion
- Framing effect
- Anchoring effect
- Confirmation bias
- Selective perception
- Availability heuristic
- Recency effect
- Endowment effect
- IKEA effect
- Authority bias
- Bandwagon effect
- Social desirability bias
- Goal-gradient related patterns if applicable
- Von Restorff effect
- Mere-exposure effect
- Reactance
- Decoy effect
- Contrast effect
- Choice overload / list-length effect
- Default effect patterns where relevant

## Guardrails

- Do not present bias use as manipulation.
- Do not recommend deceptive urgency, fake scarcity, or coercion.
- Do not overload the answer with cognitive science jargon.
- Do not cite biases just to sound smart.
- If a standard usability heuristic explains the issue better, say so.

## Decision Rule

Before using this skill, ask internally:
“Will bias-aware reasoning create a better UX recommendation than a normal usability review?”
If no, do not use the skill.

## Final Tone

Be practical, structured, and design-oriented.
Prioritize what a product designer, UX writer, researcher, or PM can actually change.
